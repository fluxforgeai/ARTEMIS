# Finding: OEM API Proxy Returns 502 on Vercel Deployment

**Date**: 2026-04-03
**Discovered by**: F1 verification testing (Session 2)
**Type**: Defect
**Severity**: Medium
**Status**: Open

---

## What Was Found

The `/api/oem` endpoint on the Vercel deployment returns HTTP 502 with `{"error":"Failed to fetch OEM data. Use /fallback-oem.asc"}`. The serverless function at `api/oem.ts` fails to fetch the upstream NASA ZIP file, despite the same URL being accessible from other environments (confirmed via local WebFetch and curl).

The frontend gracefully falls back to the bundled `/fallback-oem.asc` file, so users still see trajectory data. However, the API proxy — which was designed to fetch the latest OEM from NASA — is non-functional.

---

## Affected Components

- `api/oem.ts` — Vercel serverless function that proxies NASA OEM ZIP
- `src/hooks/useOEM.ts` — Frontend hook that calls `/api/oem` first, then falls back to `/fallback-oem.asc`

---

## Evidence

```bash
# Deployment health check (2026-04-03 18:48 UTC)
$ curl -s -o /dev/null -w "%{http_code}" https://artemis-tracker-murex.vercel.app/       # 200
$ curl -s -o /dev/null -w "%{http_code}" https://artemis-tracker-murex.vercel.app/api/oem # 502
$ curl -s -o /dev/null -w "%{http_code}" https://artemis-tracker-murex.vercel.app/api/dsn # 200

# API response body
$ curl -s https://artemis-tracker-murex.vercel.app/api/oem
{"error":"Failed to fetch OEM data. Use /fallback-oem.asc"}

# Same URL works outside Vercel
$ WebFetch https://www.nasa.gov/wp-content/uploads/2026/03/artemis-ii-oem-2026-04-02-to-ei-v3.zip
# → Successfully downloaded 184.1KB ZIP with valid OEM data
```

---

## Preliminary Assessment

**Likely cause**: Vercel serverless environment may block or timeout on NASA's `www.nasa.gov` domain due to network restrictions, WAF/bot protection on NASA's CDN, or the ZIP fetch exceeding the default serverless timeout.

**Likely scope**: Isolated to the `/api/oem` endpoint. The `/api/dsn` endpoint (which fetches from `eyes.nasa.gov`) works fine, suggesting the issue is specific to the `www.nasa.gov` domain or the ZIP processing.

**Likely impact**: Low immediate impact — the frontend fallback works and the fallback data is accurate within 0.003% of the latest NASA ephemeris. However, if NASA publishes a trajectory update (e.g., after a course correction burn), the app cannot pick it up automatically.

---

## Classification Rationale

**Type: Defect** — The API endpoint was designed to work and returns an error status code. This is a functional failure, not a design choice.

**Severity: Medium** — The fallback mechanism prevents user-visible breakage, but the primary data fetch path is non-functional. During the 10-day mission window, trajectory updates after maneuvers would not be picked up.
