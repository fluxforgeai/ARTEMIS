## Findings

### Warnings

**W1** | `src/hud/ProgressBar.tsx:17-38` | useMemo dependency recomputes every second unnecessarily
- **Agent**: Efficiency Sentinel
- **Impact**: Moderate — sort + map of 18 milestones runs every 1s tick via `elapsedHours` dependency, but milestone status only changes ~18 times across the entire mission
- **Current**: `useMemo(..., [elapsedHours])` where `elapsedHours` changes every second
- **Suggested**: Quantize to whole hours (`Math.floor(elapsedHours)`) for the memo dependency, or pre-sort MILESTONES at module level to reduce per-recompute work

### Suggestions

**S1** | `src/data/mission-config.ts:2-3` | Hours should be the primary constant, not days
- **Agent**: Paradigm Enforcer, Efficiency Sentinel
- **Issue**: `MISSION_DURATION_DAYS = 217.53 / 24` then `MISSION_DURATION_HOURS = MISSION_DURATION_DAYS * 24` round-trips through division/multiplication. The canonical value from NASA is in hours (217.53).
- **Suggested**: Make `MISSION_DURATION_HOURS = 217.53` the primary constant and derive `MISSION_DURATION_DAYS = MISSION_DURATION_HOURS / 24`.

**S2** | `src/data/mission-config.ts:27` + `src/hud/ProgressBar.tsx:19` | MILESTONES should be exported pre-sorted
- **Agent**: DS&A Reviewer, Efficiency Sentinel
- **Issue**: MILESTONES is defined in chronological order but consumers (ProgressBar, MissionEventsPanel) defensively sort with `[...MILESTONES].sort()`. Pre-sorting eliminates duplicated work and maintenance risk.
- **Suggested**: Either guarantee sort order via code comment + lint rule, or export a `SORTED_MILESTONES` constant.

**S3** | `src/hud/ProgressBar.tsx:42` | Float equality for milestone lookup is fragile
- **Agent**: DS&A Reviewer, Efficiency Sentinel
- **Issue**: `milestoneData.findIndex(m => m.missionElapsedHours === externalHoveredHours)` uses `===` on floating-point values. Safe today (both sides originate from same literal), but fragile if any intermediate code ever computes/rounds the value.
- **Suggested**: Pass milestone index or name instead of raw float for cross-component coordination.

**S4** | `src/hud/ProgressBar.tsx:7` | Redundant alias for imported constant
- **Agent**: Paradigm Enforcer
- **Issue**: `const TOTAL_MISSION_HOURS = MISSION_DURATION_HOURS` is a no-op alias. Now that the value is imported from config, the local alias adds indirection with no benefit.
- **Suggested**: Use `MISSION_DURATION_HOURS` directly at the usage site (line 22).

**S5** | `src/data/mission-config.ts:1,4` | Mutable Date objects as module-level constants
- **Agent**: Paradigm Enforcer
- **Issue**: `LAUNCH_EPOCH` and `MISSION_END_EPOCH` are `Date` objects, which are mutable. Any consumer could call `.setTime()` and corrupt shared state.
- **Suggested**: `Object.freeze()` on both, or export as epoch milliseconds and construct `Date` at call sites.
