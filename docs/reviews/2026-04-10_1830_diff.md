## Executive Summary

1 critical, 2 warnings, 9 suggestions across 13 files. The virtual clock architecture is sound — DataDriver with 60Hz interpolation and 4Hz store throttle is the right design. The critical finding is a circular import between the store and useAlerts via `clearFiredMilestones()`. The two warnings relate to milestone alerts being silently skipped at high replay rates, and dual-write clock drift at 10,000x. Suggestions are optimization opportunities for replay performance.

**Verdict: BLOCKED** -- 1 critical finding must be resolved before shipping.

## Critical Findings

### C1: Side effect in Zustand setter creates circular import

- **File**: `src/store/mission-store.ts:144`
- **Agent**: Paradigm Enforcer
- **Issue**: `setTimeMode` calls `clearFiredMilestones()` imported from `src/hooks/useAlerts.ts` as an imperative side effect before calling `set()`. This creates a circular dependency: the store imports from useAlerts, and useAlerts imports from the store. Zustand actions should be pure state transitions. The hidden coupling makes the store depend on a hook module.
- **Impact**: Circular import may cause initialization order issues. Violates Zustand convention of pure setters. Hidden coupling between store and hook.
- **Suggested approach**: Move `clearFiredMilestones()` call to a `subscribe` listener in `useAlerts.ts` that reacts to `timeControl.mode` changes, or call it from the TimeControls component before invoking `setTimeMode`.

## Warnings

### W1: 30s milestone check interval misses milestones at high replay rates

- **File**: `src/hooks/useAlerts.ts:77-99`
- **Agent**: Complexity Analyst, DS&A Reviewer
- **Issue**: The milestone check runs on a fixed 30s wall-clock interval. At 10,000x replay, 30 wall-seconds = 83 simulated hours. The T-30min alert window (0.5 simulated hours) passes in 180ms of wall time — the 30s interval will almost never sample within that window. Most milestones will be silently skipped during fast replay.
- **Suggested approach**: Drive milestone checks from DataDriver's 4Hz store-sync tick (where `simEpochMs` is already known), or scale the interval inversely with playback rate (e.g., `Math.max(1000, 30_000 / rate)`).

### W2: Dual-write clock drift — ref vs store diverge up to 41 min at 10,000x

- **File**: `src/components/DataDriver.tsx:19,30-41,91`
- **Agent**: DS&A Reviewer
- **Issue**: DataDriver's `simTimeRef` advances every frame (60Hz) but only syncs to the store at 4Hz. At 10,000x rate, 250ms wall time = 2,500 seconds (41.7 min) of simulated time drift. The spacecraft appears in the "future" segment of the trajectory because Trajectory reads the store (lagging) while the 3D position reads the ref (leading).
- **Suggested approach**: Either batch `setSimTime` into the same `set()` call as `setSpacecraft` (already at 4Hz), or expose the high-frequency time via a shared ref that Trajectory can read without React re-renders.

## Suggestions

### S1: Trajectory useMemo recomputes 3,239-vector loop at 4Hz during replay

- **File**: `src/components/Trajectory.tsx:57-94`
- **Agent**: Complexity Analyst, Efficiency Sentinel
- **Issue**: `simEpochMs` changes at 4Hz, recomputing the full OEM loop + `splitAroundBodies` every 250ms. The past/future boundary only moves when crossing an OEM vector (~4 min intervals). At 1x this is 960x more frequent than needed.

### S2: DataDriver issues 3 separate Zustand set() calls per 4Hz tick

- **File**: `src/components/DataDriver.tsx:78-99`
- **Agent**: Efficiency Sentinel
- **Issue**: `setSpacecraft`, `setSimTime`, `setMoonPosition` each trigger separate subscriber notifications. Batch into a single `set()` to reduce notifications from 12/sec to 4/sec.

### S3: `tickSimulatedTime` is dead code — no callers

- **File**: `src/store/mission-store.ts:163-175`
- **Agent**: DS&A Reviewer
- **Issue**: The action is defined in the interface and implemented but never called. DataDriver performs the tick logic locally. Future developers may call it thinking it's canonical, creating a conflicting tick source.

### S4: URL sync subscribe fires on all store changes, not just timeControl

- **File**: `src/hooks/useTimeControlInit.ts:35`
- **Agent**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel
- **Issue**: `useMissionStore.subscribe()` fires on every `set()` (12+/sec). Use selector-based subscribe: `subscribe((s) => s.timeControl, callback, { equalityFn: shallow })`.

### S5: Clamping logic duplicated across 3 places

- **File**: `src/components/DataDriver.tsx:34-38`, `src/store/mission-store.ts:155,165`
- **Agent**: Complexity Analyst
- **Issue**: `Math.max(LAUNCH_EPOCH, Math.min(..., MISSION_END_EPOCH))` appears in DataDriver, `setSimTime`, and `tickSimulatedTime`. Consolidate to one location.

### S6: Moon position new object every 4Hz forces Moon re-render

- **File**: `src/components/DataDriver.tsx:94-99`, `src/components/Moon.tsx:11`
- **Agent**: Efficiency Sentinel
- **Issue**: Fresh `{ x, y, z }` object each tick defeats reference equality. Moon moves slowly (6-hour ephemeris intervals), causing ~4 unnecessary re-renders/sec. Compare values before `setMoonPosition`.

### S7: useMission creates new Date on every 4Hz tick

- **File**: `src/hooks/useMission.ts:7`
- **Agent**: Efficiency Sentinel
- **Issue**: `new Date(simEpochMs)` is allocated at 4Hz but `getMissionElapsed` only calls `.getTime()`. Accept numeric epoch directly.

### S8: ProgressBar useMemo triggers at 4Hz for 18 milestones

- **File**: `src/hud/ProgressBar.tsx` (transitive via useMission)
- **Agent**: Complexity Analyst
- **Issue**: `elapsedHours` changes at 4Hz but milestone statuses only change ~18 times across the mission. Quantize to whole hours for the memo dependency.

### S9: DSN interval keeps running during sim/replay (harmless no-ops)

- **File**: `src/hooks/useDSN.ts:11-12`
- **Agent**: Complexity Analyst, DS&A Reviewer
- **Issue**: The 30s interval fires and checks `mode !== 'live'`, returning immediately. Negligible cost but could tear down/re-establish the interval on mode change.
