## Warnings

### W1: AlertsBanner timer management is fragile

**File**: `src/hud/AlertsBanner.tsx:13-31`
**Agents**: Complexity Analyst, DS&A Reviewer, Paradigm Enforcer, Efficiency Sentinel

The `useEffect` computes `visible = alerts.slice(0, 3)` separately from the render body (which computes it again at line 45). Timer cleanup only runs on re-render, not on individual dismiss — so timers for manually dismissed alerts linger until the next `alerts` state change. Additionally, if an alert leaves the visible window (pushed to position 3+), its timer continues running.

**Suggested fix**: Extract `visible` into a `useMemo`. Clear timers immediately in the dismiss handler. Reconcile the full timer map against `visible` in the effect body, not just cleanup.

---

### W2: useAlerts subscribes to entire spaceWeather object

**File**: `src/hooks/useAlerts.ts:25`
**Agents**: Efficiency Sentinel

The selector `useMissionStore((s) => s.spaceWeather)` returns a new object reference on every `setSpaceWeather` call. The effect fires on every 5-second weather update even when only non-alert-relevant fields change (e.g., `solarWindSpeed`, `solarWindDensity`). This also causes HUD (which calls `useAlerts`) to re-render unnecessarily.

**Suggested fix**: Subscribe to specific fields (`radiationZone`, `kpIndex`, `activeEvents`) as separate selectors. Or move alert hooks into a dedicated `<WeatherAlertDriver />` component that renders null.

---

### W3: useAlerts milestone checking coupled to spaceWeather + impure time source

**File**: `src/hooks/useAlerts.ts:62-65`
**Agents**: Complexity Analyst, Paradigm Enforcer, DS&A Reviewer

Milestone approach notifications depend on elapsed time (`Date.now()`), not space weather data, yet run inside the `spaceWeather` effect. This couples unrelated concerns and means milestones are checked every 5s as a side effect of weather updates. The `Date.now()` call is impure relative to the effect's declared dependency.

**Suggested fix**: Split milestone checking into its own `useEffect` with a dedicated interval (e.g., every 30s). Derive MET from store state or a stable clock rather than `Date.now()`.

---

### W4: useAlerts firedMilestones ref never cleared

**File**: `src/hooks/useAlerts.ts:72-79`
**Agents**: Paradigm Enforcer

The `firedMilestones` Set accumulates entries indefinitely. If the component remounts (React Strict Mode, hot reload), the ref reinitializes and prior milestone alerts are lost, potentially causing duplicate alerts.

**Suggested fix**: Scope `firedMilestones` at module level if it should persist across remounts, or add cleanup logic in the effect return.

---

### W5: SpaceWeatherPanel uses 4 separate Zustand selectors

**File**: `src/hud/SpaceWeatherPanel.tsx:33-36`
**Agents**: Complexity Analyst, Paradigm Enforcer, Efficiency Sentinel

Four independent `useMissionStore` calls each register separate subscriptions. On every store mutation, 4 equality checks run. Since all values come from `spaceWeather`, a single selector is sufficient.

**Suggested fix**: `const { kpIndex, solarWindSpeed, radiationZone, source } = useMissionStore((s) => s.spaceWeather);` — one subscription, one equality check.

---

### W6: addAlert dedup has subtle race condition

**File**: `src/store/mission-store.ts:106-109`
**Agents**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel

The dedup check scans the alerts array with `.some()` and uses `Date.now()` inside the Zustand `set()` callback. Two alerts arriving in the same millisecond could both pass the dedup check. The `Date.now()` also makes the reducer impure.

**Suggested fix**: Use a dedicated `Map<string, number>` keyed by `type+message` for O(1) dedup lookup. Use the alert's `timestamp` field instead of `Date.now()` for comparison.
