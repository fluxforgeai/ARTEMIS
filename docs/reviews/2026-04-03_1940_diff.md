## Warnings

### W1: Redundant memory copies in ZIP extraction
**File**: `api/oem.ts:59,63`
**Agents**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel
**Issue**: `bytes.slice()` allocates a new `Uint8Array` copy, then `Buffer.from()` on line 63 copies it again. Two unnecessary allocations of the compressed payload (~188KB each). On the stored path (line 59), `bytes.slice()` also copies unnecessarily since `TextDecoder.decode()` accepts a view.
**Suggested approach**: Replace `bytes.slice()` with `bytes.subarray()` (zero-copy view). For the deflate path, use `inflateRawSync(bytes.subarray(dataOffset, dataOffset + compressedSize))` directly — `inflateRawSync` accepts `Uint8Array`.

### W2: No fetch timeout / abort signal
**File**: `api/oem.ts:15`
**Agent**: Efficiency Sentinel
**Issue**: The upstream `fetch()` to nasa.gov has no timeout. If NASA is slow, the function hangs until Vercel kills it, returning a generic 504 instead of a meaningful error.
**Suggested approach**: Pass `{ signal: AbortSignal.timeout(8000) }` to `fetch()` so the function can return a useful error before Vercel terminates it.

### W3: Wildcard CORS
**File**: `api/oem.ts:7`
**Agent**: Efficiency Sentinel
**Issue**: `Access-Control-Allow-Origin: *` allows any origin to call this endpoint. While OEM data is public, the proxy could be abused for bandwidth amplification (triggering repeated NASA fetches through the Vercel function).
**Suggested approach**: Restrict to the known deployment origin or add rate limiting. Note: this pattern is consistent across all API endpoints (`dsn.ts`, `horizons.ts`, `donki.ts`), so a fix should be applied project-wide.

### W4: Inconsistent error logging in catch clause
**File**: `api/oem.ts:32`
**Agent**: Paradigm Enforcer
**Issue**: The catch block binds the error (`catch (err)`) and logs with `console.error`, while all other API handlers use bare `catch {}` with no logging.
**Suggested approach**: Either adopt `console.error` logging across all handlers (preferred for debugging) or remove it here for consistency. The OEM handler's approach is arguably better practice.

## Suggestions

### S1: Use DataView for binary integer reads
**File**: `api/oem.ts:43-47,81,85-86`
**Agents**: DS&A Reviewer, Efficiency Sentinel
**Issue**: Manual bitwise shifts for little-endian reads (`bytes[i] | (bytes[i+1] << 8) | ...`) have a sign-extension risk: `bytes[i+3] << 24` produces a signed 32-bit integer. If byte has bit 7 set, the result goes negative. `DataView.getUint16/getUint32(offset, true)` returns unsigned values and is more readable.
**Suggested approach**: Create a `DataView` from the buffer and use `getUint16(offset, true)` and `getUint32(offset, true)` for all multi-byte reads.

### S2: Optimize EOCD scan with fixed-offset check
**File**: `api/oem.ts:79`
**Agents**: DS&A Reviewer, Efficiency Sentinel
**Issue**: The backward scan for the EOCD signature starts at `bytes.length - 22` and scans to 0. For ZIPs without comments (the common case), EOCD is always at the fixed offset `bytes.length - 22`. Also, no upper bound on scan distance — a malformed input forces a full O(n) scan.
**Suggested approach**: Check the fixed offset first, then fall back to a bounded scan (max 65,536 bytes from end, covering the max ZIP comment size).

### S3: Consider more aggressive caching
**File**: `api/oem.ts:11`
**Agent**: Efficiency Sentinel
**Issue**: `s-maxage=300` means the Vercel CDN re-fetches from NASA every 5 minutes. OEM trajectory data updates on the order of hours or days.
**Suggested approach**: Increase `s-maxage` to 1800-3600 and add `stale-if-error=86400` to serve stale data when NASA is down.

### S4: Extract ZIP helpers for testability
**File**: `api/oem.ts:37`
**Agent**: Paradigm Enforcer
**Issue**: 35 lines of low-level ZIP byte parsing is substantially more complex than helpers in other API files. Extracting to a separate utility would improve testability and keep the handler file focused on the proxy pattern.
**Suggested approach**: Move `extractFirstFileFromZip` and `readSizeFromCentralDirectory` to a `lib/zip.ts` utility module.
