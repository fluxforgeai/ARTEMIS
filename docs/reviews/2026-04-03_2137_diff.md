## Critical Findings

### C1: XSS via dangerouslySetInnerHTML on AI-generated content
**File**: `src/chat/ChatMessage.tsx:20,52`
**Agents**: Paradigm Enforcer, Efficiency Sentinel
**Issue**: `renderMarkdown()` applies regex-based markdown-to-HTML conversion and injects via `dangerouslySetInnerHTML`. Input comes from Gemini API responses. The function performs no sanitization and preserves any HTML tags in the input. If the model returns `<script>`, `<img onerror=...>`, or similar payloads, they execute in the user's browser.
**Suggested approach**: Sanitize output with DOMPurify before injection: `dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(renderMarkdown(text)) }}`. Apply to both the `parts` path (line 20) and fallback `message.text` path (line 52).

## Warnings

### W1: API key passed as URL query parameter
**File**: `api/chat.ts:123,143`
**Agents**: Paradigm Enforcer, Efficiency Sentinel
**Issue**: Gemini API key is passed as `?key=${apiKey}` in the URL. Query parameters appear in server logs, CDN caches, and error traces.
**Suggested approach**: Use `x-goog-api-key` HTTP header instead if supported. Otherwise document as accepted risk.

### W2: No input validation or size limit on messages array
**File**: `api/chat.ts:215-221`
**Agent**: Efficiency Sentinel
**Issue**: The `messages` array from `req.body` is forwarded to Gemini with no validation on message count, individual text length, or total payload size. An attacker could inflate API token usage or cause timeouts.
**Suggested approach**: Cap `messages.length` (e.g., last 20), cap individual `text` length (e.g., 2000 chars).

### W3: Duplicated and drifting video data
**File**: `api/chat.ts:20-28` vs `src/data/artemis-videos.ts:6-17`
**Agents**: DS&A Reviewer, Paradigm Enforcer
**Issue**: `CURATED_VIDEOS` (server, 7 entries) and `ARTEMIS_VIDEOS` (client, 10 entries) contain overlapping data with different entry counts and titles. Data has already drifted.
**Suggested approach**: Extract to a shared JSON data file or consolidate into one location.

### W4: ChatPart type divergence between server and client
**File**: `api/chat.ts:106-109` vs `src/hooks/useChat.ts:4-9`
**Agent**: Paradigm Enforcer
**Issue**: Server-side `ChatPart` uses `[key: string]: unknown` index signature (effectively untyped). Client-side uses a proper discriminated union. These independent definitions can drift.
**Suggested approach**: Tighten the server-side interface or share types via a common definition.

### W5: Unsanitized videoId in iframe src
**File**: `src/chat/ChatVideo.tsx:12`
**Agent**: Efficiency Sentinel
**Issue**: `part.videoId` interpolated directly into iframe `src` URL. If corrupted, could load an unintended URL.
**Suggested approach**: Validate `videoId` matches `/^[a-zA-Z0-9_-]{11}$/` before rendering.
