## Warnings

### W1: Per-frame vector allocations in follow-orion useFrame
**File**: `src/components/CameraController.tsx:228-232`
**Agents**: Complexity Analyst, Efficiency Sentinel
**Issue**: In `follow-orion` mode, every animation frame creates new `THREE.Vector3` objects via `new THREE.Vector3(sc.x, sc.y, sc.z)`, `bbox.normal.clone()`, and `.multiplyScalar()`. At 60fps this produces ~120-240 short-lived objects per second, increasing GC pressure.
**Suggested approach**: Hoist reusable scratch vectors into `useRef` and use `.set()` / `.copy()` instead of `new` / `.clone()` inside the frame loop.

### W2: Module-level mutable cache without invalidation (cachedPlaneNormal)
**File**: `src/components/CameraController.tsx:21`
**Agents**: Complexity Analyst, Paradigm Enforcer
**Issue**: `let cachedPlaneNormal` is a module-level singleton that is never invalidated. If OEM data changes (trajectory file reloads every 5 min via useOEM), the cache returns stale values. This pattern breaks the project's React-lifecycle conventions where other components use `useMemo` for derived data.
**Suggested approach**: Tie cache to OEM data identity (e.g., `oemData.length` or first epoch) so it invalidates on reload.

### W3: Module-level mutable cache without invalidation (cachedBBox)
**File**: `src/components/CameraController.tsx:49`
**Agents**: Complexity Analyst, Paradigm Enforcer
**Issue**: Same stale-cache risk as W2. `let cachedBBox` persists across OEM data reloads with no invalidation path.
**Suggested approach**: Same as W2 — add cache key validation.

## Suggestions

### S1: Fold flyby-point scan into computeTrajectoryBBox
**File**: `src/components/CameraController.tsx:196-199`
**Agents**: Complexity Analyst, DS&A Reviewer, Paradigm Enforcer
**Issue**: Moon-view linear scan to find max-distance point duplicates the O(n) iteration in `computeTrajectoryBBox`. The same computation also exists in `Moon.tsx:18-25`.
**Suggested approach**: Compute flyby point inside `computeTrajectoryBBox`'s single pass and cache it alongside center/range/normal.

### S2: Use typed ref instead of `any` for OrbitControls
**File**: `src/components/CameraController.tsx:109`
**Agent**: Paradigm Enforcer
**Issue**: `useRef<any>` is the only `any` in the component layer. All other refs are properly typed.
**Suggested approach**: Type as `useRef<{ target: THREE.Vector3 }>(null)` to match actual usage.

### S3: Document imperative DOM pattern in CameraDebug
**File**: `src/components/CameraDebug.tsx:17-24`
**Agent**: Paradigm Enforcer
**Issue**: Direct `document.createElement` + `document.body.appendChild` bypasses React's declarative model. This was intentional to avoid drei `<Html>` issues, but diverges from every other component.
**Suggested approach**: Add a `// DEVIATION:` comment, or refactor to use `createPortal` to document.body for React-lifecycle compliance.

### S4: Extract useIsMobile to shared hooks
**File**: `src/components/CameraController.tsx:9-18`
**Agent**: Paradigm Enforcer
**Issue**: `useIsMobile` is a general-purpose utility hook defined inline. Project convention is single-responsibility component files.
**Suggested approach**: Move to `src/hooks/useIsMobile.ts` for reuse.

### S5: Inline bounding-box widest-axis search
**File**: `src/components/CameraController.tsx:75-89`
**Agent**: DS&A Reviewer
**Issue**: 3-element array with clones to find widest axis. Since there are exactly 3 fixed axes, this could be done with three inline comparisons and no array or clones.
**Suggested approach**: Replace with three inline dot/length comparisons. Minor — runs once (cached).
