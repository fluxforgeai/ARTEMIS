## Warnings

### W1: Cumulative 31s worst-case latency across retry chain

- **File**: `api/chat.ts:150-178`
- **Agent**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel
- **Issue**: The retry chain performs up to 3 sequential HTTP requests (grounded + fallback + transient retry), each with a 10s timeout, plus a 1s sleep. Worst case: 10s + 10s + 1s + 10s = 31s wall-clock before the serverless function responds. This exceeds Vercel Hobby plan's 10s default (fits within Pro plan's 60s). The client-side fetch in `useChat.ts:51` has no timeout, so the UI spinner could block for the full 31s.
- **Suggested approach**: Consider reducing per-request timeout to 5s (total worst-case ~16s), or add a client-side `AbortSignal.timeout(20_000)` on the fetch in `useChat.ts:51`. Alternatively, use a shared `AbortController` with a single budget across all retries.

## Suggestions

### S1: Fixed 1s delay suboptimal for 429 rate-limit responses

- **File**: `api/chat.ts:167`
- **Agent**: Complexity Analyst, Efficiency Sentinel
- **Issue**: The transient retry uses a fixed 1s delay for all error codes including 429 (rate-limited). A 429 response typically includes a `Retry-After` header; 1s may be too short for rate limits to clear, wasting the retry. For 500/503 the 1s delay is appropriate.

### S2: Repeated JSON.stringify(baseBody) up to 3 times per request

- **File**: `api/chat.ts:154,162,172`
- **Agent**: Efficiency Sentinel
- **Issue**: `JSON.stringify(baseBody)` is called up to 3 times for the same object (grounding, fallback, retry). For 20 messages at 2000 chars each, this is ~40KB serialized 3 times. Could pre-serialize once and reuse.

### S3: Unconsumed response body on non-OK fetch leaks socket

- **File**: `api/chat.ts:157`
- **Agent**: Efficiency Sentinel
- **Issue**: When the first fetch (with grounding) returns non-OK, the response body is never consumed before issuing the next fetch. In Node.js (undici), unconsumed response bodies keep the socket allocated. Call `await response.text()` or `response.body?.cancel()` before the fallback fetch to release the connection.

### S4: CURATED_VIDEOS array is mutable at runtime (pre-existing)

- **File**: `api/chat.ts:24-33`
- **Agent**: Paradigm Enforcer
- **Issue**: `CURATED_VIDEOS` is declared `const` but the array is mutable. In an FP codebase, static lookup tables should use `as const` or `ReadonlyArray` for deep immutability.

### S5: VALID_ROLES is a mutable Set (pre-existing)

- **File**: `api/chat.ts:122`
- **Agent**: Paradigm Enforcer
- **Issue**: `new Set()` is mutable. Could use `as const` tuple with `.includes()` or `Object.freeze(new Set(...))` for immutability consistency.

### S6: Intent/regex mapping could use Record map (pre-existing)

- **File**: `api/chat.ts:6-8`
- **Agent**: Paradigm Enforcer
- **Issue**: The four intent-detection regexes and the `detectIntent` if-chain could be refactored into a `Record<Intent, RegExp>` map for more compositional FP style and easier extensibility.

### S7: ChatPart type duplicated between server and client (pre-existing)

- **File**: `src/hooks/useChat.ts:6-12`, `api/chat.ts:114-120`
- **Agent**: Paradigm Enforcer
- **Issue**: The `ChatPart` union type is duplicated in both files. A shared types module would prevent drift between server and client definitions.
