## Warnings

### W1: Message sanitization filter runs over unbounded input before slice

- **File**: `api/chat.ts:226-229`
- **Agent**: Complexity Analyst
- **Issue**: The `.filter().slice(-20).map()` chain creates intermediate arrays. The `.filter()` runs over the entire unbounded input `messages` array before `.slice(-20)` truncates. An attacker could send thousands of messages to force unnecessary iteration.
- **Suggested approach**: Reorder to `messages.slice(-20).filter(...).map(...)` to bound iteration upfront, or add a hard cap (e.g., reject if `messages.length > 100`) at the validation step.

## Suggestions

### S1: Regex literals in detectIntent recompiled per call

- **File**: `api/chat.ts:9-16`
- **Agent**: DS&A Reviewer, Efficiency Sentinel
- **Issue**: 4 regex objects are created on every call to `detectIntent`. Hoisting to module-level `const` avoids per-request compilation overhead.

### S2: Set instantiated per request for role validation

- **File**: `api/chat.ts:225`
- **Agent**: DS&A Reviewer
- **Issue**: `new Set(['user', 'model'])` is instantiated inside the handler on every request. Hoist to module scope as `const VALID_ROLES = new Set(...)`.

### S3: findCuratedVideo uses nested linear scan

- **File**: `api/chat.ts:30-36`
- **Agent**: DS&A Reviewer, Paradigm Enforcer
- **Issue**: Imperative `for...of` with nested `.some()` and `.includes()`. A declarative `.find()` call would be more consistent with the FP style used elsewhere. At 7 entries this is cosmetic.

### S4: ChatPart type duplicated between api/ and src/

- **File**: `api/chat.ts:106-111`
- **Agent**: Paradigm Enforcer
- **Issue**: The discriminated union is defined identically in `api/chat.ts` and `src/hooks/useChat.ts`. Due to the Vercel serverless constraint (api/ can't import from src/), a shared `types/` directory or cross-referencing comments would reduce drift risk.

### S5: buildSystemPrompt recomputes Date math per request

- **File**: `api/chat.ts:66-101`
- **Agent**: Efficiency Sentinel
- **Issue**: Date math, string formatting, and `toLocaleString()` run per request. Output changes meaningfully once per minute. A minute-granularity cache would save ~0.5ms per request under load.

### S6: generateImage uses imperative accumulator pattern

- **File**: `api/chat.ts:155-159`
- **Agent**: Paradigm Enforcer
- **Issue**: `for...of` loop pushes onto a mutable `parts` array. A `flatMap` would be more idiomatic FP.

### S7: buildChartParts uses let-mutation pattern

- **File**: `api/chat.ts:182-191`
- **Agent**: Paradigm Enforcer
- **Issue**: Declares `chartType` and `title` with `let` and conditionally reassigns. A ternary or object lookup would be more declarative.

### S8: renderMarkdown regexes recreated per render

- **File**: `src/chat/ChatMessage.tsx:8-16`
- **Agent**: DS&A Reviewer, Paradigm Enforcer, Efficiency Sentinel
- **Issue**: 6 regex objects created per call, combined with `let`-mutation chain. Hoisting regexes to module scope and chaining `.replace()` directly would avoid recompilation and improve readability.

### S9: ChatMessage not wrapped in React.memo

- **File**: `src/chat/ChatMessage.tsx:36-57`
- **Agent**: Efficiency Sentinel
- **Issue**: ChatMessage re-renders on every parent render (new messages added). Since messages are immutable once created, `React.memo` would prevent redundant DOMPurify + markdown processing for unchanged messages.

### S10: videoId regex recreated per render

- **File**: `src/chat/ChatVideo.tsx:8`
- **Agent**: DS&A Reviewer
- **Issue**: Regex `/^[a-zA-Z0-9_-]{11}$/` is compiled on every render. Hoisting to a module-level `const` is trivial.
