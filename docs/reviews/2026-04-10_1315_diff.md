## Suggestions

### S1: Retry sleep not deadline-aware — can waste up to 3s on a doomed retry

- **File**: `api/chat.ts:155-189`
- **Agent**: Complexity Analyst, DS&A Reviewer
- **Issue**: The `retryAfter` sleep (1-3s) is not aware of the remaining deadline budget. If 13s have elapsed by the time the sleep begins, the 3s sleep leaves 0s for the final fetch, which will abort immediately. The retry is futile but the user waits the full 15s+3s. Could skip the retry if remaining budget < 2s.

### S2: Unconsumed response body on non-transient error path

- **File**: `api/chat.ts:190`
- **Agent**: Efficiency Sentinel
- **Issue**: When `isTransient()` returns false, execution falls through to the `throw` at line 190 without cancelling the response body from the fetch at line 168. The socket is held until GC. Add `await response.body?.cancel()` before the throw to match the pattern at lines 166 and 181.

### S3: `isTransient()` could use a Set for maintainability

- **File**: `api/chat.ts:135`
- **Agent**: DS&A Reviewer
- **Issue**: The chained `===` comparisons work fine for 3 values, but if more transient codes are added (502, 504), a `const TRANSIENT_CODES = new Set([429, 500, 503])` with `.has()` would be more maintainable. Minor — fine as-is at 3 values.

### S4: `response.body?.cancel()` could theoretically block on slow streams

- **File**: `api/chat.ts:164-166`
- **Agent**: Complexity Analyst
- **Issue**: Awaiting `response.body?.cancel()` on a slow-streaming response could take non-trivial time. In practice, Gemini error responses are small JSON — negligible. Awareness only, no action needed.
